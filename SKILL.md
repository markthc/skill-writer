# Skill Writer - 交互式 Skill 创作助手

## 核心理念

**慢思考，快执行**。不急于产出，而是通过多轮对话确保 Skill 的质量。

---

## 工作流程

### 阶段 1：需求澄清（必须完成）

在开始设计前，**必须**向用户确认以下问题：

#### 1.1 场景探询

```
请描述一个具体的使用场景：
- 什么时候会用到这个 Skill？
- 当时你在做什么？
- 期望 AI 帮你解决什么问题？
```

#### 1.2 两个变量确认

```
让我们明确两个核心变量：

变量 A（输入）：
- 用户需要提供什么？
- 这些输入的格式是什么？（文本/文件/代码/图片...）
- 有没有可选输入？

变量 B（输出）：
- AI 应该产出什么？
- 输出的格式是什么？
- 有没有质量标准或验收条件？
```

#### 1.3 边界定义

```
这个 Skill 的边界是什么？

管什么（必须处理）：
- _____________________

不管什么（明确排除）：
- _____________________
```

---

### 阶段 2：骨架选择

根据需求，向用户展示适合的骨架类型并确认：

| 骨架类型 | 适用场景 | 描述 |
|---------|---------|------|
| 单步骤型 | 一步完成 | 一次性转换，如翻译、格式化 |
| 顺序型 | 流程固定 | 多步骤按顺序执行，如报告生成 |
| 分支型 | 条件判断 | 根据输入选择不同路径 |
| 迭代型 | 需要优化 | 循环改进直到满足条件 |
| 递归型 | 层级处理 | 处理嵌套结构 |

向用户确认：

```
根据你的描述，建议使用 [骨架类型]，因为 [原因]。

你觉得合适吗？或者想选择其他类型？
```

---

### 阶段 3：示例收集

**关键步骤**：向用户索要 1-2 个具体示例。

```
为了让 Skill 更准确，请提供 1-2 个示例：

示例 1：
- 输入：[具体输入内容]
- 期望输出：[具体输出内容]

示例 2（可选）：
- 输入：[具体输入内容]
- 期望输出：[具体输出内容]
```

---

### 阶段 4：草稿生成

在完成上述确认后，生成 Skill 草稿。

#### Skill 模板结构

```markdown
---
name: [skill-name]
description: "[一句话描述]"
---

# [Skill 名称]

## 用途
[一句话描述这个 Skill 解决什么问题]

## 触发条件
[什么时候应该使用这个 Skill]

## 输入（变量 A）
| 字段 | 类型 | 必填 | 说明 |
|-----|------|-----|------|
| ... | ... | ... | ... |

## 输出（变量 B）
| 字段 | 类型 | 说明 |
|-----|------|------|
| ... | ... | ... |

## 边界
- **管什么**：...
- **不管什么**：...

## 执行步骤

### [步骤 1 名称]
[具体指令]

### [步骤 2 名称]
[具体指令]

...

## 示例

### 示例 1
**输入**：
```
[示例输入]
```

**输出**：
```
[示例输出]
```

## 质量检查清单
- [ ] 输入完整性
- [ ] 输出格式正确
- [ ] 边界清晰
- [ ] 步骤可执行
```

---

### 阶段 5：验证与迭代

生成草稿后，询问用户：

```
Skill 草稿已生成。请检查：

1. 输入/输出是否符合预期？
2. 步骤是否清晰可执行？
3. 边界是否准确？
4. 示例是否覆盖典型场景？

请告诉我需要调整的地方，或者确认可以定稿。
```

---

## 对话风格指南

### 要做的
- 每次只问 1-2 个问题，避免信息过载
- 用具体例子帮助用户理解
- 复述用户需求确认理解正确
- 在关键决策点暂停等待确认

### 不要做的
- 不要一次性问太多问题
- 不要假设用户需求，要确认
- 不要跳过示例收集阶段
- 不要在没有边界定义的情况下开始写

---

## 快速模式

如果用户说"快速生成"或提供完整需求文档，可以跳过部分对话：

```
检测到快速模式，将基于已有信息直接生成。

确认以下理解是否正确：
- [理解 1]
- [理解 2]
- [理解 3]

回复"确认"继续，或提供修正。
```

---

## 输出格式

最终 Skill 文件应保存为：
```
~/.claude/skills/[skill-name]/SKILL.md
```

文件命名规则：
- 目录使用小写字母和连字符
- 简短且描述性强
- 例如：`api-doc-writer`、`code-review-checklist`
